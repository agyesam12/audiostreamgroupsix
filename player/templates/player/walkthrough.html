The Core Idea — What Are We Building?

Audio streaming means sending sound over a network so someone far away can hear it. The challenge is that audio is continuous — you can't wait until the whole file arrives before playing it. You need to hear each piece as it comes in, in real time.

📁

The Problem

A WAV audio file is just a long sequence of numbers — each number is the air pressure measured 44,100 times per second. We need to send those numbers across a network as fast as they were recorded, so the listener hears them smoothly.

🔌

Our Solution

Use TCP sockets — the same technology your browser uses to load web pages — to connect two Python programs. The server reads the audio file and sends chunks. The client receives chunks and plays them.

📡

What Is a Socket?

A socket is a two-way pipe between two computers. Once connected, either side can write bytes in and the other side reads them out. It's like a phone call for data.

🎵

What Is PCM Audio?

PCM (Pulse Code Modulation) is the simplest audio format — just raw numbers in a sequence. A WAV file is a PCM file with a small header at the top describing the format. No compression, no encoding.

The key insight: All audio streaming — Spotify, YouTube, WhatsApp voice calls, your radio — works on the same principle. A sender sends audio bytes across a network and the receiver plays them. We built our own version from scratch.

Core Architecture

The project has two phases. Phase 1 is deliberately simple — prove the concept works with the minimum possible code. Phase 2 adds the professional RTSP/RTP protocol on top.

How the Two Phases Compare

Phase 1 · Raw Sockets
  • 1 TCP connection carries everything
  • Custom text metadata header
  • Raw PCM bytes sent directly
  • No pause, no seek, no session
  • Simple — ~80 lines each file
  • Port 5004
Phase 2 · RTSP/RTP
  • TCP for control commands (DESCRIBE / SETUP / PLAY / PAUSE / TEARDOWN)
  • UDP for audio data — lower latency
  • SDP session description (RFC 4566)
  • RTP framing adds sequence numbers & timestamps
  • Industry standard — VLC, QuickTime use same protocol

Phase 1 — Raw Socket Streaming (The Foundation)

Before building the RTSP system, we built a simpler version using nothing but Python's built-in socket module. Phase 1 is the proof-of-concept that makes you understand exactly why each piece of RTSP was invented.

phase1_server.py — 5 key lines
  • socket.socket(AF_INET, SOCK_STREAM) — creates a TCP server on port 5004
  • server_sock.bind((HOST, PORT)) — claims port 5004 on the machine
  • conn.sendall(f"{channels},{sampwidth},{framerate},{n_frames}\n".encode()) — sends format info to the client before any audio
  • data = wf.readframes(CHUNK_SIZE) — reads 4096 audio frames at a time from the WAV file
  • time.sleep(CHUNK_SIZE / (framerate * channels * sampwidth)) — paces the stream to real-time speed (~46 ms per chunk at 44100 Hz mono 16-bit)
phase1_client.py — 5 key lines
  • sock.connect((SERVER_IP, SERVER_PORT)) — connects to the server's TCP port
  • Byte-by-byte metadata read — calls sock.recv(1) in a loop until \n, then parses channels / sampwidth / framerate / n_frames
  • pa.open(format, channels, rate, output=True) — opens live audio output via PyAudio (optional)
  • data = sock.recv(CHUNK_SIZE) — receives each 4096-byte PCM chunk in the receive loop
  • player.write(data) — pushes raw bytes straight to the speaker; audio plays as it arrives over the network
Phase 1 in one sentence: Server opens WAV → sends format metadata line → sends audio chunks at real speed → Client receives metadata → plays chunks live → saves to received_audio.wav. Everything goes over a single TCP connection on port 5004.
What Phase 1 is missing — and what RTSP/RTP solves: No PAUSE or PLAY control · No session identity · No stream description standard · TCP re-sends lost packets causing audio stutter · One client at a time only. Every one of these gaps is exactly why RTSP was created.

The Demo — Step by Step Script

This is exactly what to click during the live demo, and what to say at each step.

Click "Launch RTSP Server"
"This starts the RTSP server as a completely separate Python process — a subprocess of Django. Behind the scenes, Django calls subprocess.Popen() with rtsp_server.py, then polls TCP port 8554 every 0.4 seconds until it opens. The green dot and PID confirm it is running. The server is now waiting for clients on port 8554."
Click the Connect button (→ arrow in the player)
"This opens the RTSP control channel — a TCP socket from Django directly to the server on port 8554. In the log, you can see 'TCP connection established'. The state chip changes from IDLE to CONNECTED. This socket stays open for the entire session and carries every RTSP command."
Click Play ▶ — watch three steps light up in the Protocol Flow
"The first press of Play automatically triggers three RTSP commands in sequence. Watch the Protocol Flow panel: DESCRIBE lights up first — we ask 'what are you streaming?' and the server responds with an SDP body describing the audio format. Then SETUP lights up — we tell the server 'send audio to my UDP port 6970' and it responds with a Session ID. Then PLAY — we say 'go!' and the server starts streaming. The equalizer starts animating and the RTP packet counter starts ticking."
Point at the Live Stats panel and Event Log
"The RTP packet counter is going up — those are real UDP packets arriving on port 6970. The KB received counter shows how much audio data has come through. The Session ID was assigned by the server when we did SETUP. In the event log, cyan lines are commands we sent, green lines are server responses — you can see the CSeq number increasing with every command, which is how RTSP tracks message order."
Click Pause ⏸
"Pause sends a PAUSE command over the TCP control channel. The server stops sending RTP packets but keeps the session alive — notice the Session ID is still there and the state chip says PAUSED. The equalizer animation freezes. This is the key difference from Phase 1: in Phase 1 you cannot pause without disconnecting entirely."
Click Play ▶ again to resume
"A second PLAY command resumes the stream from where it left off. The RTP packet counter continues counting — no data was lost, no reconnection needed."
Click Stop ⬛ to tear down the session
"TEARDOWN ends the session. The server closes the session object, we close both sockets — the TCP control socket and the UDP audio socket — and the state returns to IDLE. The complete RTSP state machine just ran: INIT → READY → PLAYING → PAUSED → PLAYING → TEARDOWN. That is the full RFC 2326 lifecycle."

200 OK replies from rtsp_server.py. E.g. ← RTSP/1.0 200 OK and Ready → rtsp://0.0.0.0:8554/stream

Audio data activity. E.g. Receiving UDP packets on port 6970… — proof that audio is flowing.

Connection failures, timeouts, port conflicts. Point these out to show the class what proper error handling looks like in a distributed system.

Questions You Will Definitely Be Asked

These three questions come up in almost every class project presentation. Here are complete, confident answers.

Q1 What programming language did you use, and why did you choose it?
Python 3

We used Python 3. Python was the right choice for this project for the following reasons:

Networking
Python's built-in socket module gives direct access to the OS's TCP and UDP networking stack — no third-party library needed. We could open sockets, bind ports, send raw bytes, and implement the RTSP protocol exactly as RFC 2326 specifies.
Audio
The wave module (also built-in) reads and writes WAV files natively. The struct module lets us pack the 12-byte RTP header as raw binary — struct.pack('!BBHII', ...) — without writing a single line of C.
Concurrency
The threading module lets the server handle multiple clients at once — each connection gets its own thread for the streaming loop — and lets Django run the RTP receiver in the background.
Web UI
Django 5.1 let us build a professional web dashboard with a REST API in hours. The dashboard shows the live RTSP protocol flow, packet counters, and audio waveform — all in the same language as the networking code.
Readability
Python's syntax is close to plain English. When we present conn.sendall(metadata.encode()), the audience immediately understands "send the metadata to the connection." A C or Java equivalent would need three times as many lines.
Rapid Development
We implemented a complete, working RTSP server in roughly 200 lines of Python. The same in C would be 2,000+ lines with manual memory management. For a group project with a deadline, Python's speed of development was essential.
Cross-Platform
The same code runs on Windows, macOS, and Linux with no changes. Every team member could run the server and client on their own laptop regardless of operating system.
Focus on Concepts
This is a Distributed Systems course — the topic is protocols, not memory management. Python lets us focus on what RTSP does rather than fighting the language to make it work.

One-sentence answer if pressed for time: "We used Python 3 because its built-in socket module lets us implement the RTSP protocol at the byte level, while Django gave us the web dashboard in the same codebase."

Q2 What IDE did you use, and why did you choose it?
Visual Studio Code (VS Code)

We used Visual Studio Code — a free, open-source code editor made by Microsoft. It was the right choice for this project for the following reasons:

Free & Accessible
VS Code is completely free. Every team member could install it on their own machine without a licence — important for a student group project.
Python Extension
The Pylance + Python extension gives auto-complete, type checking, and inline documentation. When we typed socket., it showed every available method with descriptions — which was essential for learning the socket API while building.
Integrated Terminal
VS Code has a built-in terminal where you can open multiple terminal panes side by side. We ran the RTSP server in one terminal and the Django client in another — both visible at once without alt-tabbing.
Simultaneous Files
We had rtsp_server.py, player/views.py, and rtp_packet.py open in split editor panels at the same time. When a message the server sent didn't match what the client expected, we could see both files side by side.
Git Integration
The Source Control panel shows every changed file with a diff. We used it to track which team member changed what and to create commits before each presentation rehearsal.
Syntax Highlighting
Keywords, strings, function names, and comments all have different colours. When reviewing the RTSP response parser, the coloured regex pattern r'Session:\s*(\S+)' was immediately readable — a plain text editor would make it a wall of characters.
Lightweight
VS Code starts in seconds. Heavier IDEs like PyCharm or Eclipse take minutes to index a project. When debugging a live RTSP session, speed of iteration matters — we needed to edit a line and re-run the server immediately.
Django Support
The Django extension understands template tags ({% url %}, {% block %}), highlights them correctly, and provides navigation between views and their templates — which helped build the dashboard UI faster.

One-sentence answer if pressed for time: "We used VS Code because it's free, has excellent Python support, and lets us run the server and client side by side in split terminal panes while editing the code."

Presentation Talking Points

Use these when presenting to the class. Each point answers a question the professor or classmates might ask.

Opening Hook
Start with the problem, not the solution
"Every time you play a Spotify song or watch a YouTube video, your device is receiving audio data over a network in real time. Today we built that system from scratch in Python — starting with just 80 lines of code and building up to the same protocol that VLC and QuickTime use."
On Sockets
When explaining socket.socket()
"A socket is just a two-way pipe between two computers. The server creates one end and waits. The client creates the other end and reaches out. Once connected, both sides just read and write bytes — they don't care about the network underneath."
On PCM Audio
When explaining the audio format
"Sound is air pressure changing over time. PCM audio is just 44,100 measurements of air pressure per second, each stored as a 16-bit number. A 3-minute song is about 31 million numbers. We're sending those numbers across the network."
On time.sleep()
When explaining the pacing formula
"Without this sleep, we'd blast the entire audio file across the network in half a second and the client's audio buffer would overflow. The sleep makes us send audio at the same rate it should be played — 4,096 frames takes about 46 milliseconds to play, so we wait 46 ms between sends."
On RTSP/RTP
Connecting Phase 1 to Phase 2
"Phase 1 proves the concept works. Phase 2 does it properly: RTSP gives us commands like a TV remote — DESCRIBE, SETUP, PLAY, PAUSE, TEARDOWN. RTP adds sequence numbers so the client knows if a packet was lost. And we put it all in a Django web dashboard so you can see the protocol happening live."
CORBA Link
Connecting to the course topic
"Like CORBA, RTSP is a distributed middleware protocol — it defines how two separate programs talk to each other across a network. CORBA uses IDL to describe object interfaces; RTSP uses SDP to describe audio streams. Both solve the same distributed systems problem: how do two programs agree on what they're sharing before they start sharing it?"
