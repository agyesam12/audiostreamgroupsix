The Core Idea — What Are We Building?

Audio streaming means sending sound over a network so someone far away can hear it. The challenge is that audio is continuous — you can't wait until the whole file arrives before playing it. You need to hear each piece as it comes in, in real time.

📁

The Problem

A WAV audio file is just a long sequence of numbers — each number is the air pressure measured 44,100 times per second. We need to send those numbers across a network as fast as they were recorded, so the listener hears them smoothly.

🔌

Our Solution

Use TCP sockets — the same technology your browser uses to load web pages — to connect two Python programs. The server reads the audio file and sends chunks. The client receives chunks and plays them.

📡

What Is a Socket?

A socket is a two-way pipe between two computers. Once connected, either side can write bytes in and the other side reads them out. It's like a phone call for data.

🎵

What Is PCM Audio?

PCM (Pulse Code Modulation) is the simplest audio format — just raw numbers in a sequence. A WAV file is a PCM file with a small header at the top describing the format. No compression, no encoding.

The key insight: All audio streaming — Spotify, YouTube, WhatsApp voice calls, your radio — works on the same principle. A sender sends audio bytes across a network and the receiver plays them. We built our own version from scratch.

How the Two Phases Compare

Phase 1 · Raw Sockets
  • 1 TCP connection carries everything
  • Custom text metadata header
  • Raw PCM bytes sent directly
  • No pause, no seek, no session
  • Simple — ~80 lines each file
  • Port 5004
Phase 2 · RTSP/RTP
  • TCP for control commands (DESCRIBE / SETUP / PLAY / PAUSE / TEARDOWN)
  • UDP for audio data — lower latency
  • SDP session description (RFC 4566)
  • RTP framing adds sequence numbers & timestamps
  • Industry standard — VLC, QuickTime use same protocol

Phase 1 — Raw Socket Streaming (The Foundation)

Before building the RTSP system, we built a simpler version using nothing but Python's built-in socket module. Phase 1 is the proof-of-concept that makes you understand exactly why each piece of RTSP was invented.

phase1_server.py — 5 key lines
  • socket.socket(AF_INET, SOCK_STREAM) — creates a TCP server on port 5004
  • server_sock.bind((HOST, PORT)) — claims port 5004 on the machine
  • conn.sendall(f"{channels},{sampwidth},{framerate},{n_frames}\n".encode()) — sends format info to the client before any audio
  • data = wf.readframes(CHUNK_SIZE) — reads 4096 audio frames at a time from the WAV file
  • time.sleep(CHUNK_SIZE / (framerate * channels * sampwidth)) — paces the stream to real-time speed (~46 ms per chunk at 44100 Hz mono 16-bit)
phase1_client.py — 5 key lines
  • sock.connect((SERVER_IP, SERVER_PORT)) — connects to the server's TCP port
  • Byte-by-byte metadata read — calls sock.recv(1) in a loop until \n, then parses channels / sampwidth / framerate / n_frames
  • pa.open(format, channels, rate, output=True) — opens live audio output via PyAudio (optional)
  • data = sock.recv(CHUNK_SIZE) — receives each 4096-byte PCM chunk in the receive loop
  • player.write(data) — pushes raw bytes straight to the speaker; audio plays as it arrives over the network
Phase 1 in one sentence: Server opens WAV → sends format metadata line → sends audio chunks at real speed → Client receives metadata → plays chunks live → saves to received_audio.wav. Everything goes over a single TCP connection on port 5004.
What Phase 1 is missing — and what RTSP/RTP solves: No PAUSE or PLAY control · No session identity · No stream description standard · TCP re-sends lost packets causing audio stutter · One client at a time only. Every one of these gaps is exactly why RTSP was created.

System Architecture — How All Pieces Connect

The project has three distinct layers that each have a different job. Understanding how they connect is the key to explaining the architecture to anyone — professor, classmate, or interviewer.

Why This Three-Layer Design?

🌐

Browser → Django (HTTP)

The browser doesn't speak RTSP — it speaks HTTP. Every button click becomes a simple POST request to a Django REST endpoint. Django translates those into RTSP messages on the control socket.

🔧

Django as RTSP Client

player/views.py holds module-level globals (_rtsp_sock, _state, _cseq). Django's dev server runs as one process, so these globals persist across HTTP requests — that is how session state survives between button clicks.

⚙️

Server as Subprocess

rtsp_server.py is a completely independent Python process — spawned by Django with subprocess.Popen(). It has its own memory, its own sockets, its own threads. Django can stop it, restart it, or crash it without affecting the web UI.

📡

TCP control + UDP audio

Commands (DESCRIBE/SETUP/PLAY/PAUSE/TEARDOWN) use TCP because they must arrive correctly. Audio uses UDP because a lost audio packet is better skipped than waited for — delay is worse than a tiny glitch in music.

_wait_for_port() — the polling readiness check
def _wait_for_port( host, port, timeout=20): deadline = time.time() + timeout while time.time() < deadline: try: s = socket.socket( socket.AF_INET, socket.SOCK_STREAM) s.settimeout(0.8) s.connect((host, port)) s.close() return True except OSError: time.sleep(0.4) return False
Lines 52–64
Busy-polls a TCP port until it accepts connections or the deadline passes.
deadline = time.time() + 20 — sets an absolute point in time 20 seconds from now.
s.settimeout(0.8) — each connection attempt waits at most 0.8 s before giving up (so the port-closed error is fast).
s.connect((host, port)) — attempts a real TCP connection. If the server process hasn't called listen() yet, this raises an OSError (connection refused).
s.close() — immediately closes the test socket. We don't need it; we just needed to confirm the port was reachable.
time.sleep(0.4) — wait 0.4 s between attempts. Total worst case: 25 polls × 0.4 s + 25 × 0.8 s = ~30 s, but the 20 s deadline cuts it off.
Returns True if the port opened, False if the 20 s passed without success.
_send_rtsp() — builds and sends every RTSP message
Part 1 — Build the request
def _send_rtsp( method, extra_headers=None): global _cseq, _rtsp_sock if not _rtsp_sock: return None, 'Not connected' _cseq += 1 url = f'rtsp://{RTSP_IP}:{RTSP_PORT}/stream' lines = [ f'{method} {url} RTSP/1.0', f'CSeq: {_cseq}', 'User-Agent: Group6RTSPPlayer/1.0', ] if _state.get('session_id'): lines.append( f"Session: {_state['session_id']}") if extra_headers: for k, v in extra_headers.items(): lines.append(f'{k}: {v}') lines += ['', '']
Lines 67–83
Constructs a valid RTSP/1.0 request from a method name:
global _cseq — declares that this function modifies the module-level counter.
_cseq += 1 — RTSP requires every request to have a monotonically increasing CSeq number. This is how the server matches responses to requests.
The lines list builds headers one by one. Each header is a key-value pair.
if _state.get('session_id') — after SETUP, every subsequent request must include the Session header. Without it, the server won't know which session to operate on.
lines += ['', ''] — appending two empty strings means '\r\n'.join(lines) produces ...last-header\r\n\r\n — the mandatory double-CRLF that tells the server the header block is finished.
Part 2 — Send and receive the response
_rtsp_sock.sendall( '\r\n'.join(lines).encode()) data = b'' _rtsp_sock.settimeout(15) while b'\r\n\r\n' not in data: chunk = _rtsp_sock.recv(4096) if not chunk: break data += chunk
Lines 85–92
Sends the request and reads the response headers:
sendall() guarantees every byte is sent — unlike send() which may only send part of the data on a busy network.
.encode() converts the Python string to bytes (UTF-8) because sockets only transmit bytes.
settimeout(15) — if the server doesn't respond within 15 seconds, raise a timeout exception rather than blocking forever.
while b'\r\n\r\n' not in data — keeps reading until the response header block terminator is seen. This is the loop that was broken before the double-CRLF fix: SETUP responses without \r\n\r\n would make this loop hang until the 15-second timeout.
Part 3 — Read the response body (for DESCRIBE)
m = re.search( rb'Content-Length:\s*(\d+)', data) if m: cl = int(m.group(1)) bstart = data.find(b'\r\n\r\n') + 4 remain = cl - (len(data) - bstart) while remain > 0: chunk = _rtsp_sock.recv(remain) if not chunk: break data += chunk remain -= len(chunk) return data.decode('utf-8', errors='ignore'), None
Lines 94–107
Reads the SDP body after the headers for DESCRIBE responses:
re.search(rb'Content-Length:\s*(\d+)', data) — uses a regex on raw bytes to find the Content-Length header value. The rb prefix means "raw bytes regex".
bstart = data.find(b'\r\n\r\n') + 4 — finds where the body starts (4 bytes after the double-CRLF).

Key Technical Concepts — What to Know Cold

The RTSP State Machine

RTSP defines a strict state machine. You cannot PLAY before SETUP, and you cannot SETUP before DESCRIBE. The server enforces this — an out-of-order command returns a 4xx error.

The RTP Packet — 12-Byte Header + Audio Payload

Every audio packet sent from the server follows the RFC 3550 RTP format. Our rtp_packet.py builds and parses this header using struct.pack('!BBHII', ...).

PT = 96 (Dynamic): Payload types 0–95 are statically assigned by IANA (e.g. PT=0 = G.711 µ-law, PT=8 = G.711 A-law). PT=96–127 are dynamic — you define what they mean in the SDP. Our SDP says a=rtpmap:96 L16/44100/1, which means "PT 96 = raw 16-bit PCM, 44100 Hz, 1 channel."

What the SDP Body Actually Looks Like

When you send DESCRIBE, the server responds with a 200 OK and an SDP body that describes everything about the stream. Here is what our server sends and what each line means:

v=0 o=- 0 0 IN IP4 0.0.0.0 s=Group6 Audio Stream t=0 0 m=audio 6970 RTP/AVP 96 a=rtpmap:96 L16/44100/1
v=Protocol version. Always 0 for SDP.
o=Origin. Username, session ID, version, network type, address type, address. The - means anonymous.
s=Session name. Human-readable name for this stream — "Group6 Audio Stream."
t=Timing. Start and stop times as NTP timestamps. 0 0 = permanent / no fixed schedule.
m=Media description. audio type, port 6970, transport RTP/AVP, payload type 96. This is the core line — it tells the client where and how the audio will arrive.
a=Attribute — codec mapping. Defines what PT 96 means: L16 (raw 16-bit PCM), 44100 Hz sample rate, 1 channel (mono). Without this, the client wouldn't know how to decode the packets.
Why SDP matters for the presentation: SDP is what makes RTSP interoperable. If you replaced our custom server with a VLC RTSP server, our client could still work — it would just read the SDP and adapt. That is the power of a standard protocol over Phase 1's custom metadata line.

Architecture Decision — Why Pre-Convert to WAV?

MP3 is a compressed format. The RTSP server streams raw PCM audio (uncompressed 16-bit samples at 44,100 Hz). Converting MP3 to PCM takes a few seconds. If we converted every time someone clicked Play, there would be a noticeable delay before streaming started.

Old approach (broken)
  • Click "Launch Server" → run miniaudio to decode MP3 → write sample.wav
  • If miniaudio fails, silently fall back to a synthesised test tone
  • Only sample.wav exists — switching songs means re-running a slow decode
  • rtsp_server.py looked for ../sample.wav (wrong path)
New approach (working)
  • Run python convert_songs.py once → produces puul.wav, do_better.wav, biggest_nathaniel.wav
  • Song switching = instant file copy (no decode)
  • rtsp_server.py now looks for sample.wav in its own directory (correct path)
  • Errors are surfaced clearly instead of falling back to a tone

Questions You Will Definitely Be Asked

These three questions come up in almost every class project presentation. Here are complete, confident answers.

Q1 What programming language did you use, and why did you choose it?
Python 3

We used Python 3. Python was the right choice for this project for the following reasons:

Networking
Python's built-in socket module gives direct access to the OS's TCP and UDP networking stack — no third-party library needed. We could open sockets, bind ports, send raw bytes, and implement the RTSP protocol exactly as RFC 2326 specifies.
Audio
The wave module (also built-in) reads and writes WAV files natively. The struct module lets us pack the 12-byte RTP header as raw binary — struct.pack('!BBHII', ...) — without writing a single line of C.
Concurrency
The threading module lets the server handle multiple clients at once — each connection gets its own thread for the streaming loop — and lets Django run the RTP receiver in the background.
Web UI
Django 5.1 let us build a professional web dashboard with a REST API in hours. The dashboard shows the live RTSP protocol flow, packet counters, and audio waveform — all in the same language as the networking code.
Readability
Python's syntax is close to plain English. When we present conn.sendall(metadata.encode()), the audience immediately understands "send the metadata to the connection." A C or Java equivalent would need three times as many lines.
Rapid Development
We implemented a complete, working RTSP server in roughly 200 lines of Python. The same in C would be 2,000+ lines with manual memory management. For a group project with a deadline, Python's speed of development was essential.
Cross-Platform
The same code runs on Windows, macOS, and Linux with no changes. Every team member could run the server and client on their own laptop regardless of operating system.
Focus on Concepts
This is a Distributed Systems course — the topic is protocols, not memory management. Python lets us focus on what RTSP does rather than fighting the language to make it work.

One-sentence answer if pressed for time: "We used Python 3 because its built-in socket module lets us implement the RTSP protocol at the byte level, while Django gave us the web dashboard in the same codebase."

When explaining the audio format

On time.sleep()
When explaining the pacing formula
"Without this sleep, we'd blast the entire audio file across the network in half a second and the client's audio buffer would overflow. The sleep makes us send audio at the same rate it should be played — 4,096 frames takes about 46 milliseconds to play, so we wait 46 ms between sends."
