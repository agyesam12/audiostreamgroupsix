The Core Idea — What Are We Building?

Audio streaming means sending sound over a network so someone far away can hear it. The challenge is that audio is continuous — you can't wait until the whole file arrives before playing it. You need to hear each piece as it comes in, in real time.

📁

The Problem

A WAV audio file is just a long sequence of numbers — each number is the air pressure measured 44,100 times per second. We need to send those numbers across a network as fast as they were recorded, so the listener hears them smoothly.

🔌

Our Solution

Use TCP sockets — the same technology your browser uses to load web pages — to connect two Python programs. The server reads the audio file and sends chunks. The client receives chunks and plays them.

📡

What Is a Socket?

A socket is a two-way pipe between two computers. Once connected, either side can write bytes in and the other side reads them out. It's like a phone call for data.

🎵

What Is PCM Audio?

PCM (Pulse Code Modulation) is the simplest audio format — just raw numbers in a sequence. A WAV file is a PCM file with a small header at the top describing the format. No compression, no encoding.

The key insight: All audio streaming — Spotify, YouTube, WhatsApp voice calls, your radio — works on the same principle. A sender sends audio bytes across a network and the receiver plays them. We built our own version from scratch.

Core Architecture

The project has two phases. Phase 1 is deliberately simple — prove the concept works with the minimum possible code. Phase 2 adds the professional RTSP/RTP protocol on top.

How the Two Phases Compare

Phase 1 · Raw Sockets
  • 1 TCP connection carries everything
  • Custom text metadata header
  • Raw PCM bytes sent directly
  • No pause, no seek, no session
  • Simple — ~80 lines each file
  • Port 5004
Phase 2 · RTSP/RTP
  • TCP for control commands (DESCRIBE / SETUP / PLAY / PAUSE / TEARDOWN)
  • UDP for audio data — lower latency
  • SDP session description (RFC 4566)
  • RTP framing adds sequence numbers & timestamps
  • Industry standard — VLC, QuickTime use same protocol

🎵 generate_sample_wav() — creates a test tone if no audio file exists
def generate_sample_wav(filename): import struct import math
Lines 25–31
Function definition and two more imports.
struct converts Python numbers into raw bytes (e.g., an integer into 2 bytes). We need this to write audio samples into the WAV file.
math gives us math.sin() to generate a smooth sine wave tone. These imports are inside the function because they're only needed here.
sample_rate = 44100 duration = 5 frequency = 440.0 num_samples = sample_rate * duration amplitude = 32767
Lines 33–37
The maths behind audio:
44 100 samples per second is the CD standard — how many times per second we measure the air pressure.
440 Hz is the musical note A4. The sine wave completes 440 full cycles per second.
num_samples = 44100 × 5 = 220 500 — we need 220,500 numbers to represent 5 seconds of audio.
32767 is the maximum value for a 16-bit signed integer — the loudest the audio can get without distortion.
with wave.open(filename, 'w') as wf: wf.setnchannels(1) wf.setsampwidth(2) wf.setframerate(sample_rate)
Lines 39–42
Create and configure the WAV file:
wave.open(filename, 'w') creates a new WAV file. The 'w' means "write mode."
setnchannels(1) → mono audio (one channel). Stereo would be 2.
setsampwidth(2) → 2 bytes per sample = 16-bit audio. 16 bits gives 65,536 possible loudness levels.
setframerate(44100) → 44,100 samples per second tells any player how fast to play back the samples.
frames = [] for i in range(num_samples): sample = int(amplitude * math.sin(2 * math.pi * frequency * i / sample_rate)) frames.append( struct.pack('<h', sample))
Lines 44–47
Generate 220,500 audio samples mathematically:
For each sample number i from 0 to 220,499:
— math.sin(2π × 440 × i / 44100) produces a value between −1.0 and +1.0, cycling 440 times per second.
— Multiply by 32767 to scale it to 16-bit range.
— struct.pack('<h', sample) converts the Python integer into 2 raw bytes in little-endian format — exactly what a WAV file expects.
The < means little-endian (least significant byte first), h means signed short (16-bit integer).
wf.writeframes(b''.join(frames)) print(f"[SERVER] Generated: {filename}")
Lines 49–51
b''.join(frames) glues all 220,500 two-byte chunks into one long byte string — b means bytes not text.
wf.writeframes() writes all those bytes into the WAV file in one go.
The file is automatically closed when the with block ends.
The print() logs a message to the terminal so we know it worked.
📤 stream_audio() — the heart of the server: reads WAV and sends it to the client
def stream_audio(conn, addr): print(f"[SERVER] Client: {addr}") if not os.path.exists(AUDIO_FILE): generate_sample_wav(AUDIO_FILE)
Lines 54–60
Called once for each client that connects.
conn is the socket object — a two-way pipe to the client.
addr is a tuple like ('192.168.1.5', 43201) — the client's IP and port.
os.path.exists() checks if the WAV file is on disk. If not, we generate a test tone first so the server always has something to stream.
with wave.open(AUDIO_FILE, 'rb') as wf: channels = wf.getnchannels() sampwidth = wf.getsampwidth() framerate = wf.getframerate() n_frames = wf.getnframes()
Lines 62–66
Open the audio file and read its technical details.
'rb' = read binary — we're reading raw bytes, not text.
The wave module reads the WAV header and extracts:
channels → 1 (mono) or 2 (stereo)
sampwidth → bytes per sample (2 for 16-bit)
framerate → 44100 samples/second
n_frames → total number of audio frames
We need these numbers to tell the client how to decode what follows.
metadata = ( f"{channels},{sampwidth}," f"{framerate},{n_frames}\n" ) conn.sendall(metadata.encode())
Lines 71–72
The "handshake" — tell the client how to decode the audio before sending it.
Example: "1,2,44100,220500\n" — one line of comma-separated values followed by a newline character (\n).
The client reads this first line and uses those four numbers to set up its audio player.
.encode() converts the Python string into bytes, because sockets only send bytes.
sendall() guarantees every byte is sent — unlike send() which might only send part of it.
chunk_num = 0 while True: data = wf.readframes(CHUNK_SIZE) if not data: break
Lines 75–79
The main streaming loop — runs until the audio file ends.
while True: loops forever — the break statements inside are the only way out.
wf.readframes(4096) reads the next 4096 audio frames from the WAV file. Each frame is sampwidth × channels bytes — for mono 16-bit audio, each frame is 2 bytes, so we read 8,192 bytes at a time.
if not data: break — when we reach the end of the file, readframes() returns an empty byte string (b''), which is falsy in Python.
try: conn.sendall(data) chunk_num += 1 time.sleep(CHUNK_SIZE / (framerate * channels * sampwidth))
Lines 81–86
Send one chunk and pace the stream:
conn.sendall(data) sends the raw audio bytes to the client over TCP.
The time.sleep() is the key insight — without it, we'd send the whole file in milliseconds and the client's audio buffer would overflow. The formula 4096 ÷ (44100 × 1 × 2) ≈ 0.0464 seconds makes us send audio at exactly the rate it should be played back.
except (BrokenPipeError, ConnectionResetError): print("[SERVER] Client disconnected") break print(f"Sent {chunk_num} chunks.") conn.close()
Lines 88–93
Handle client disconnection gracefully:
If the client closes their window mid-stream, the next sendall() will raise a BrokenPipeError (Linux/Mac) or ConnectionResetError (Windows).
We catch these and break out of the loop cleanly instead of crashing.
conn.close() releases the network resources after streaming ends.
🚀 run_server() — creates the TCP socket and waits for clients
server_sock = socket.socket( socket.AF_INET, socket.SOCK_STREAM ) server_sock.setsockopt( socket.SOL_SOCKET, socket.SO_REUSEADDR, 1 )
Lines 97–98
Create and configure the server socket:
socket.AF_INET → use IPv4 addresses (e.g., 192.168.1.5).
socket.SOCK_STREAM → use TCP (reliable, ordered, connection-based). The alternative would be UDP (SOCK_DGRAM).
SO_REUSEADDR = 1 is critical — without it, if you restart the server within 60 seconds, you get "Address already in use". This option bypasses that wait.
server_sock.bind((HOST, PORT)) server_sock.listen(1)
Lines 99–100
bind((HOST, PORT)) claims port 5004. The socket now "owns" that port — like putting your name on a door. No other program can use port 5004 while this runs.
listen(1) puts the socket into "waiting" mode and creates a queue for incoming connection requests. The 1 means only 1 connection waiting in the queue at a time.
while True: conn, addr = server_sock.accept() stream_audio(conn, addr)
Lines 105–107
The accept loop — waits for clients forever:
server_sock.accept() blocks — the program pauses here and does nothing until a client connects. When one does, it returns a new socket conn (the private pipe to that client) and their address addr.
We then call stream_audio() to serve them. Note: this handles one client at a time. Phase 2 uses threads to handle multiple clients simultaneously.
if __name__ == '__main__': run_server()
Lines 110–111
The entry point guard — Python's standard way to run a file:
__name__ is a special Python variable. When you run python phase1_server.py directly, Python sets __name__ to '__main__'.
But if another file does import phase1_server, __name__ would be 'phase1_server' — so run_server() would NOT auto-execute.
This pattern is the Python convention for "only run this if the file is executed directly."

The server sends the metadata line like "1,2,44100,220500\n". We don't know how many bytes it is, so we read one byte at a time and stop when we see \n.

The receiving loop — mirrors the server's sending loop:

We use exactly the same audio format parameters (channels, sampwidth, framerate) that we received from the server in the metadata header. This guarantees the saved file matches the original.

Why We Upgraded to RTSP/RTP

Phase 1 works but has limitations that professional streaming systems need to solve. RTSP/RTP was invented specifically to solve them.

No Pause or Seek in Phase 1
Once the server starts sending, there's no way to pause or resume. RTSP adds PAUSE and PLAY commands on a separate control channel — you can pause the stream without dropping the connection.
No Session Identification
Phase 1 has no session concept — if you reconnect, the server has no idea who you are. RTSP assigns a Session: 482931 ID on SETUP, so the server tracks each client's state independently.
TCP Is Not Ideal for Audio Data
TCP re-transmits lost packets, which causes audio gaps. Phase 2 sends audio over UDP — if a packet is lost, the next one arrives on time instead of stalling. RTP adds sequence numbers so the client can detect lost packets.
No Stream Description
Phase 1's metadata is a single custom text line. RTSP uses SDP (Session Description Protocol, RFC 4566) — a standardised format that any RFC-compliant player (VLC, QuickTime, ffmpeg) can understand.
Single-Client Only
Phase 1's run_server() handles one client at a time. Phase 2 spawns a new threading.Thread for each incoming connection, allowing many clients to receive the stream simultaneously.

Q1 What programming language did you use, and why did you choose it?
Python 3

We used Python 3. Python was the right choice for this project for the following reasons:

Networking
Python's built-in socket module gives direct access to the OS's TCP and UDP networking stack — no third-party library needed. We could open sockets, bind ports, send raw bytes, and implement the RTSP protocol exactly as RFC 2326 specifies.
Audio
The wave module (also built-in) reads and writes WAV files natively. The struct module lets us pack the 12-byte RTP header as raw binary — struct.pack('!BBHII', ...) — without writing a single line of C.
Concurrency
The threading module lets the server handle multiple clients at once — each connection gets its own thread for the streaming loop — and lets Django run the RTP receiver in the background.
Web UI
Django 5.1 let us build a professional web dashboard with a REST API in hours. The dashboard shows the live RTSP protocol flow, packet counters, and audio waveform — all in the same language as the networking code.
Readability
Python's syntax is close to plain English. When we present conn.sendall(metadata.encode()), the audience immediately understands "send the metadata to the connection." A C or Java equivalent would need three times as many lines.
Rapid Development
We implemented a complete, working RTSP server in roughly 200 lines of Python. The same in C would be 2,000+ lines with manual memory management. For a group project with a deadline, Python's speed of development was essential.
Cross-Platform
The same code runs on Windows, macOS, and Linux with no changes. Every team member could run the server and client on their own laptop regardless of operating system.
Focus on Concepts
This is a Distributed Systems course — the topic is protocols, not memory management. Python lets us focus on what RTSP does rather than fighting the language to make it work.

One-sentence answer if pressed for time: "We used Python 3 because its built-in socket module lets us implement the RTSP protocol at the byte level, while Django gave us the web dashboard in the same codebase."

Presentation Talking Points

Use these when presenting to the class. Each point answers a question the professor or classmates might ask.

Opening Hook
Start with the problem, not the solution
"Every time you play a Spotify song or watch a YouTube video, your device is receiving audio data over a network in real time. Today we built that system from scratch in Python — starting with just 80 lines of code and building up to the same protocol that VLC and QuickTime use."
On Sockets
When explaining socket.socket()
"A socket is just a two-way pipe between two computers. The server creates one end and waits. The client creates the other end and reaches out. Once connected, both sides just read and write bytes — they don't care about the network underneath."
On PCM Audio
When explaining the audio format
"Sound is air pressure changing over time. PCM audio is just 44,100 measurements of air pressure per second, each stored as a 16-bit number. A 3-minute song is about 31 million numbers. We're sending those numbers across the network."
On time.sleep()
When explaining the pacing formula
"Without this sleep, we'd blast the entire audio file across the network in half a second and the client's audio buffer would overflow. The sleep makes us send audio at the same rate it should be played — 4,096 frames takes about 46 milliseconds to play, so we wait 46 ms between sends."
On RTSP/RTP
Connecting Phase 1 to Phase 2
"Phase 1 proves the concept works. Phase 2 does it properly: RTSP gives us commands like a TV remote — DESCRIBE, SETUP, PLAY, PAUSE, TEARDOWN. RTP adds sequence numbers so the client knows if a packet was lost. And we put it all in a Django web dashboard so you can see the protocol happening live."
CORBA Link
Connecting to the course topic
"Like CORBA, RTSP is a distributed middleware protocol — it defines how two separate programs talk to each other across a network. CORBA uses IDL to describe object interfaces; RTSP uses SDP to describe audio streams. Both solve the same distributed systems problem: how do two programs agree on what they're sharing before they start sharing it?"
Live Demo
What to show during the demo
Open the dashboard at localhost:8000. Click Launch Server — show the log: "Waiting for clients." Click Play — show the DESCRIBE → SETUP → PLAY sequence lighting up. Point to the RTP packet counter incrementing. Play Puul by Lasmid through the browser. Show the Protocol Flow diagram animating.
If Asked
Why Python and not C/Java?
"Python's socket module gives direct access to the OS networking stack, the same as C would. Python lets us build the protocol layer clearly without memory management overhead — the important thing is the protocol logic, not the language."
Pro tip for the demo: Have python manage.py runserver already running before the presentation. Open the dashboard in a browser tab. Have the /docs page open in another tab. Walk through both during the presentation — show the live dashboard first (the exciting part), then explain the code using the annotated walkthrough.
